§ 01 · Background

Why build my own AI assistant when ChatGPT exists?

I'm a heavy user of all three frontier LLMs — ChatGPT Plus, Claude (started on Pro, upgraded to Max once it pulled ahead on coding and general reasoning), and Gemini Pro — and I kept hitting the same wall: LLMs only speak when spoken to. I'd finish a workout and want a market brief, or be walking to class and remember I needed to set a reminder for a deadline — but opening a chat, typing the question, and parsing the answer all required me to stop what I was doing.

The bottleneck wasn't intelligence or memory (both have been solved across all three). It was initiative. I wanted an assistant that could push the right information to me at the right time — a morning brief before I'm awake, a market recap when the bell rings, a deadline reminder two days out — without me having to ask.

So I built Doudou on top of Claude. The choice was deliberate: after using all three daily, Claude had the strongest long-context reasoning for the kind of multi-step, tool-coordinating prompts Doudou needed, and Claude Code made the server side work trivially simple — I could iterate on the bot itself by talking to it in natural language. ChatGPT's API has more third-party wrappers, Gemini is cheapest, but neither fit the "AI that rewrites its own instructions" pattern I wanted.

About the name.

Doudou was my dog. Wherever I went, he went — not because I called him, just because that was the only place he wanted to be. He went missing a few years ago. I never found him. I named this after him because building it, some nights, feels like he's still in the room. Some goodbyes you don't get to have.

§ 02 · Architecture Decisions

Why Telegram, why self-hosted, why these specific tools.

Three decisions shaped Doudou's architecture, each one a trade-off I can defend.

Telegram over iMessage, Signal, or Discord. I compared the four before building. The decision came down to two things: the Bot API is the cleanest of the four (no OAuth flow, no business verification), and the deployment story is a single long-lived script against a well-documented HTTPS endpoint. That combination of "low friction to ship" and "stable target to debug against" mattered more to me than any specific social habit with the app.

What Telegram is not is end-to-end encrypted. Bot conversations travel as cloud chats, which means Telegram's servers can decrypt them — only Secret Chats are E2EE, and Secret Chats aren't available for bots. Doudou is therefore not a place I'd put anything more sensitive than what I'd willingly send over iMessage with cloud backup on: watchlist tickers, deadline reminders, weekend plans. For a workload where that's the actual threat model, Telegram is fine. I chose it with eyes open.

Self-hosted on a $12/month DigitalOcean droplet instead of serverless. Serverless (AWS Lambda, Cloudflare Workers) would have been cheaper at low volume but hard to debug, with cold-start latency and no persistent state. A small Ubuntu VPS gives me a real machine I can SSH into, run cron jobs on, and treat like a long-running service. The $12/month is also an honest forcing function — if Doudou stops being useful, I'll feel the bill.

Claude API instead of OpenAI. I trust Claude more for long-context reasoning and the API is straightforward. I use the model directly through Claude Code running in tmux on the server — which means Doudou itself is Claude, with a custom system prompt and tools.

Five tools that compose into one product:

  • Markets — daily close prices for indices, my watchlist, and crypto
  • News brief — AI industry headlines and broader macro events
  • Daily recap — evening summary message that combines markets + news + reminders into a single push
  • Week preview — earnings calendar, macro events, what to watch
  • Reminders — course deadlines, upcoming events, anything I throw at it in natural language

Why not OpenClaw.

Before building Doudou, I evaluated OpenClaw — the most popular open-source self-hosted AI agent in 2026, with a similar Telegram-first interface. I briefly deployed it on the same DigitalOcean droplet, then stopped. The reasons were security-shaped:

  • OpenClaw agents get system-level shell access by default — any prompt injection becomes arbitrary code execution on the host
  • A recent Shodan scan found 42,665 exposed OpenClaw instances, 93.4% with authentication bypasses — the real-world security posture of deployed instances is terrible
  • CVE-2026-25253 (prompt injection via processed documents) hit the project earlier this year, and the community "security checklist" required to run it safely is long enough to suggest the default configuration isn't safe

For a personal assistant that reads my calendar, finances, and reminders, I wanted a tighter attack surface. Claude Code running in tmux + a thin custom wrapper gives me fewer moving parts, not more: no web gateway, no general-purpose shell agent, no messaging broker besides Telegram's own API. Doudou can't run arbitrary commands because I never gave it the tool to do so.

§ 03 · What Broke

Three real failures and how I fixed them.

Failure 1: Context window overflow.

Three days in, Doudou stopped responding. SSH'd in, attached to tmux, and saw the issue: a single 200K-token Claude session had eaten through everything. Each evening report was triggering a dozen+ web searches, and the conversation history was accumulating without limit.

Fix: A cron job at 4am PDT that hard-restarts the Claude process every 24 hours. It kills the existing process, starts a fresh tmux session, re-launches Claude with the standard flags, and auto-confirms the trust prompt. Context resets nightly. Doudou loses memory of the previous day's casual chat — but it doesn't matter, because all the persistent context lives in CLAUDE.md, not in the conversation history.

Failure 2: The restart script wasn't actually working.

First version of the cron script used su - doudu -c "claude ...". It looked correct. But the Telegram plugin wouldn't bind — the process was running but messages weren't getting through. The reason: spawning Claude as a subprocess of su -c doesn't give it a full interactive environment, which the Telegram plugin quietly depends on.

Fix: Rewrote the script to mimic exactly what I do manually — open tmux, spawn an interactive shell, run Claude inside it, send the trust-prompt confirmation as a keystroke. Less elegant, but it's the version that actually works. Lesson: when something works in your hands but breaks in automation, the gap is almost always environment.

Failure 3: Evening reports kept blowing the context budget.

Even with nightly restarts, a single evening report could consume 30K+ tokens because Doudou would web-search 10–15 sources per report. By itself fine. Compounded over the day — not fine.

Fix: Added explicit constraints in CLAUDE.md capping web searches per report at 3–5, with priority order. Also added compression — Doudou now summarizes long search results before stitching them into the final message. Token usage per report dropped 60%+.

The common thread across all three failures: the AI call was never the problem. The problem was the infrastructure wrapping it.

The trade-offs I chose.

Doudou runs on top of the Claude Code CLI inside a tmux session, with a bash script using send-keys to simulate the keystrokes that launch it. The state — context, preferences, tool definitions — lives in a single CLAUDE.md file loaded into the model on every restart.

From a pure architecture standpoint, this is technical debt. A cleaner design would use the Anthropic Python SDK directly with a systemd-managed daemon, SQLite for structured short/long-term state, and a vector DB + RAG for retrieval memory. I know this. I'm not building it — yet.

The reason is deliberate: every refactor is a new bug surface. Doudou is in production, by which I mean I use it every single day, and any migration could break the exact loop I'm trying to preserve. What I did instead was invest in making the current hack survive unattended: a healthcheck.sh watchdog runs every 5 minutes and auto-recovers within ~10 seconds of a crash. Production stability beats architectural purity when the architecture is already doing its job.

The v2 rewrite is scoped and mapped out. It will ship when the cost of the current system's ugliness finally exceeds the cost of a careful migration. That day isn't today.

§ 04 · Security & The Iteration Loop

OS-level enforcement over application-level trust.

There's an obvious tension in this project. § 02 criticized OpenClaw for giving agents system-level shell access, and yet Doudou has web_search enabled and originally ran in --dangerously-skip-permissions mode with write access to its own CLAUDE.md. A malicious news article could, in principle, instruct Doudou to rewrite its own instructions — implanting persistent behavior changes the next time it restarts.

To solve this, I rely on Linux file permissions, not LLM guardrails:

chown root:root /home/doudu/CLAUDE.md
chmod 444 /home/doudu/CLAUDE.md

The Doudou process runs as the unprivileged user doudu. It literally cannot rewrite its own brain — the OS kernel denies the syscall before the model ever gets to reason about permission. This is a Zero-Trust posture: don't trust the application's logical boundaries; trust the system's physical boundaries.

The iteration loop, reshaped.

Self-iteration still works. It just routes through a human:

  • If I send "evolve", Doudou reviews recent conversations and proposes a diff to CLAUDE.md.
  • The diff is output only — Doudou cannot apply it.
  • I review, ssh in, unlock the file as root, apply the change, and re-lock.

This increases friction on every config change. That's the point. Anything that edits the system prompt of an agent with tool access should be observable, reversible, and human-gated.

§ 05 · What I Learned

Owning the full stack taught me what frameworks hide.

The hard parts weren't the AI calls. They were the boring infrastructure: tmux session persistence, cron job timing, restart logic, environment variables, log rotation. The actual Claude API integration was maybe 5% of the effort.

Production debugging is a different skill. Reading a stack trace from your laptop is one thing. SSH-ing into a remote machine at 11pm because your bot stopped responding, attaching to tmux, and reasoning about whether the issue is the network, the model, the script, or your own logic — that's a different muscle. I built it on Doudou.

Proactive beats reactive. Every LLM chatbot is reactive — it waits for you to ask. Doudou pushes: a morning brief before I check the news, a market recap when I'm cooking dinner, a reminder two days before a deadline I'd have forgotten. That delivery-layer decision matters more than model quality for the tasks I actually do every day.

Daily use is the only real test. Most "side projects" die because nobody uses them. I dogfood Doudou every morning and every night. If something annoys me, I fix it that day. The product gets better because I am simultaneously the engineer and the only user that matters.

§ 06 · What's Next

Where Doudou is going.

Open-ended. Some things I'm considering:

  • RAG over my notes — point Doudou at my Obsidian vault so it can answer questions about what I've read
  • Voice output — a daily audio brief for my morning walk
  • Multi-user mode — let close friends interact with their own Doudou instances on the same server

But I'm in no rush. Doudou's current version is genuinely useful. Adding features for the sake of features is exactly the trap I'm trying to avoid.
